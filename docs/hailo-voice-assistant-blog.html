Raspberry Pi 5 + Hailo AI HAT+2: Building a Local Voice Assistant the Hard Way (Because No One Wrote It Down)

A technical deep-dive from the trenches of making this actually work

For a long time, my smart home voice assistant was Alexa. And for a long time, I was living with its inconsistencies. Every command was a small act of hope — you speak, and you wait to see if it understood, if it connected, if the right thing happened. When it did not respond, there was no way to know why. Was it a network issue? A skill timeout? Did it even hear me? The black box nature of it bothered me more and more. I wanted something I could understand, debug, and control.

That frustration is what eventually brought me to running a fully local voice assistant on Raspberry Pi 5 with Home Assistant. I have been on a longer journey to remove cloud dependencies from my home — replacing various hubs and third-party services one by one — and a local voice assistant was always the hardest piece. The LLM inference was the blocker. Running even a small model on Pi's CPU was painful. You ask something, you wait. You wait more.

Then I got Hailo AI HAT+2. This is a small M.2 accelerator board that sits on top of the Pi 5. It has a dedicated NPU with 40 TOPS of compute. The promise was: run AI workloads at the edge, without cloud, without GPU, without burning your electricity bill. I was skeptical. But the results surprised me — and the path from "Hailo NPU running a local LLM" to "something that actually replaces Alexa" turned out to be a lot less direct than I expected.

Normally this post would come later in my series — after I write about removing all the cloud dependencies first. But buying bleeding-edge hardware like AI HAT+2 comes with a cost beyond the price tag: limited community support, sparse documentation, and a lot of time spent reading source code to understand what is actually happening. I wanted to publish this now, in case there are other people out there struggling with exactly this hardware and not finding answers. The offline-first setup series will catch up.

The Stack

Before I go into the problems, let me show you what we built:

Waveshare ESP32-S3 (wake word + mic) → hailo-whisper (STT, NPU) → Home Assistant → hailo-ollama (LLM, NPU) → piper (TTS) → Speaker

All the heavy lifting runs on one Pi 5 with AI HAT+2, all AI models on the Hailo-10H NPU. The ESP32-S3 handles the always-on part at the edge. Here is the complete picture:

ComponentHardware / ContainerProtocolNotes
Wake Word + MicWaveshare ESP32-S3 AI Smart SpeakerESPHome / WyomingAlways-on, runs on device
Speech-to-Texthailo-whisper containerWyomingHailo-10H NPU
LLM Inferencehailo-ollama containerHTTP (custom)Hailo-10H NPU
Text-to-Speechpiper containerWyomingCPU
Orchestrationhomeassistant containerREST / WebSocketCPU

The ESP32-S3 is the always-listening piece. It detects the wake word locally on the microcontroller itself — nothing is streamed to the Pi until the wake word fires. Once triggered, it sends the audio over Wyoming protocol to hailo-whisper for transcription. Home Assistant then takes over: sends the transcribed text to hailo-ollama, gets a response, passes it to Piper, and the audio plays back through the speaker. Clean pipeline. But making these pieces talk to each other — that is where the real engineering begins.

First Challenge: hailo-ollama Is Not Ollama

Home Assistant's Extended OpenAI Conversation integration expects standard Ollama on port 11434. hailo-ollama shares the port but almost nothing else: the /v1/models endpoint doesn't exist, the model manifest format is completely different (HEF-based, not GGUF/OCI), hailo-ollama pull crashes on invocation, and there's no /api/create — so no model aliases. Pointing HA at hailo-ollama out of the box fails silently in multiple ways.

The Solution: A Translation Proxy

We put a Python proxy between HA and hailo-ollama. It listens on 11434 (what HA expects), translates /v1/models to /api/tags, and forwards to hailo-ollama on 11436.

The harder part was JSON sanitisation — two separate bugs that interact. HA sends literal newline characters inside JSON strings, which is invalid. The proxy fixes those before parsing. But even with valid JSON, hailo-ollama's internal HailoRT library re-serialises message content and re-introduces unescaped newlines in its own pass. The second fix walks all message content after parsing and replaces newlines with spaces before forwarding. Both layers are necessary — fixing only one still crashes the pipeline.

NPU Sharing: The Detail That Catches Everyone

Both hailo-whisper and hailo-ollama need access to the same physical NPU device (/dev/h1x-0). HailoRT by default does not share well. The solution is the HAILO_VDEVICE_GROUP_ID environment variable, set to the same value in both containers:

# compose.yaml
services:
  hailo-whisper:
    environment:
      - HAILO_VDEVICE_GROUP_ID=SHARED
  hailo-ollama:
    environment:
      - HAILO_VDEVICE_GROUP_ID=SHARED
      - HAILO_OLLAMA_VDEVICE_GROUP_ID=SHARED

With this, HailoRT manages a shared VDevice pool and both containers can time-share the NPU without conflict. One important operational note: when installing a new model, hailo-whisper must be stopped first. Model download triggers VDevice initialisation in hailo-ollama, and if hailo-whisper is simultaneously holding the device, you get a hard crash.

Second Challenge: The Accuracy vs. Latency Trap

With the proxy working, the full pipeline was running end-to-end. qwen2.5:1.5b on the Hailo-10H NPU was a real improvement over CPU — roughly 6–8x faster. But "faster than CPU inference" and "fast enough to replace Alexa" turned out to be two very different bars.

The model is small and non-deterministic. On commands with an obvious single answer — "dim the office lights to 30 percent" — it occasionally returned the right tool call targeting the wrong entity, or emitted brightness_pct: 80 for "darker". The natural response to this is to add more context: more examples in the system prompt, more entity descriptions, more guidance on when to call which service.

This worked. Accuracy improved. But each example added to the system prompt grows the context window, and on a 1.5B model running on constrained hardware, context window size has a direct cost: response time climbed to around 30 seconds for a typical command with a well-filled context.

30 seconds is not a voice assistant. That is a loading screen. My original goal was better accuracy and latency compared to Alexa. Running qwen2.5:1.5b on the AI HAT+2's limited memory, I was getting better accuracy at the cost of latency that made the system unusable for daily life. So I decided to pivot.

The Pivot: Determinism Over Intelligence

While wrestling with the LLM accuracy problem, I noticed something: when Hailo Whisper correctly transcribed a short phrase that matched one of HA's built-in intents — "turn on the lights", "set brightness to fifty percent" — the response was near-instant. Under two seconds total. HA's local intent recognition fired, the light changed, Piper spoke the confirmation. No LLM involved, no latency, no hallucination. Just correct.

That observation led to a reframe: I had been feeding more and more examples into the LLM system prompt to make it handle all the commands I actually use. But if I was going to enumerate all those combinations anyway — why not add them directly to Home Assistant as custom sentence patterns, and skip the LLM entirely for those cases?

This is what Home Assistant's hassil intent system is built for. When prefer_local_intents: true is set on a pipeline, HA runs its local NLU first. If the phrase matches a pattern, the action fires deterministically in milliseconds. The LLM is only invoked when local matching fails.

The gap is that HA's built-in sentences cover common patterns like "turn on the lights" and "set brightness to N percent" but miss surface forms that people actually use. I filled these gaps with custom sentence YAML files.

Using Claude to Generate the Sentences

Writing exhaustive custom sentences by hand is tedious. I used Claude to generate them. Given a target intent and a list of entities, it produces all the natural surface forms a person might use — including the variants that are easy to miss.

For example, for bedroom light dimming, the generated sentences covered patterns like:

# lights.yaml
language: "en"
intents:
  HassLightSet:
    data:
      - sentences:
          - "dim [the] {area} [lights|light] to {brightness} [percent|%|pct]"
          - "[set] {area} [lights|light] [to] {brightness} [percent|%|pct]"
          - "{area} [lights|light] at {brightness} [percent|%|pct]"
          - "make [the] {area} [lights|light] {brightness} [percent|%|pct]"

Beyond lights, we built out sentence files for:

  • shades.yaml — 12 natural shade commands ("open the main bedroom blinds", "bedroom shades to private", "living room sunny mode") mapped to HassTurnOn with fixed name slots pointing to the correct PowerView scene friendly names
  • heat_pump.yaml — a custom HeatPumpSource intent with slots for each of the four temperature source options ("bedroom", "main bedroom", "living room", "thermostat")

The HeatPumpSource intent requires one extra step: an intent_script block in configuration.yaml that maps it to a select.select_option service call and generates the spoken confirmation. Built-in intents like HassLightSet handle their own service calls and speech; custom intents need this wiring done explicitly.

The result: the commands I actually use daily — adjust a light, move the shades, change the heat pump sensor — never touch the LLM. Response is under two seconds, and the outcome is always correct.

Handling the Imperfect Transcription Problem

Even with Hailo Whisper doing well, there were still times it did not understand me correctly. English is not my first language, and the model would sometimes transcribe "dim the lights" as "dim the lights" with a different entity name, or mishear a room name entirely.

For a set of known commands, the failure mode I cared about was not "wrong answer" but "no answer at all" — the STT output not matching any custom sentence, causing the LLM to kick in and introduce latency. What matters is getting immediate responses even for near-miss transcriptions.

The solution was adding a corrections layer to the Whisper pipeline. Before the transcription reaches HA, the proxy checks the output against a configurable corrections file — a list of regex-to-replacement pairs:

[
  ["\\bleaving room\\b", "living room"],
  ["\\bturn of\\b(?!\\s*f)", "turn off"]
]

The first entry fixes a persistent mishearing: "leaving room" instead of "living room". The second catches a common truncation where the final consonant drops off — "turn of" instead of "turn off" — but uses a negative lookahead so it does not accidentally rewrite "turn off" itself.

Corrections handle known misrecognitions and common shorthand that would not otherwise match a sentence pattern. They are cheap to add and make the system noticeably more robust for the commands you actually use.

To make it easy to grow the corrections list — and to discover what commands other family members are actually saying — I enabled Hailo Whisper to log both the transcribed sentence and the Home Assistant response it triggered. The log becomes a running record of all voice interactions: what was said, what HA understood, what it did. From that, it is straightforward to spot patterns and either add corrections or add new custom sentences for commands that are being used but not yet covered.

Automating the Corrections List with the Local LLM

Reading the log manually is fine when you first set up the system. It stops being fine after a few weeks. So we automated it: a Python script runs every night at 2AM, reads the last 24 hours of errors, asks the local LLM whether each one looks like a Whisper misrecognition, and appends any new corrections to the file.

This is, admittedly, a bandaid. A larger model on beefier hardware would not need it — it would transcribe accurately enough that the corrections list stays short. But on constrained hardware running a limited set of daily commands, this approach works well. The command vocabulary is small and stable, the error rate is low, and running inference on a tiny nightly payload costs nothing since the NPU is otherwise idle.

The transcript log format is tab-separated: timestamp, raw transcript, normalised transcript, status (action_done or error), and the HA result. The script filters for error rows from the past 24 hours and passes each one to the local LLM — qwen2.5:1.5b via the same OpenAI-compat endpoint on port 11434 — along with the full list of friendly names pulled live from HA's /api/states. That entity list is the context that makes the classification useful: the model can see that "Dining Room Lights" is a real device and reason that "dying room" is a phonetic near-miss.

One lesson learned quickly: do not ask a 1.5B model to produce a structured JSON array for multiple inputs in one call. It will invent field names, truncate the output mid-item, and apply few-shot examples to the wrong transcripts. The reliable approach is one call per transcript, asking for a single line of output:

YES: dying room -> dining room
# or
NO

Even then, the model ignores the format instruction about half the time and adds an explanation paragraph. So instead of enforcing the format, we search for the YES: X -> Y pattern anywhere in the response and treat everything else as a no.

Even with that flexibility, the model will occasionally hallucinate a correction — applying an example from the few-shot prompt to a transcript where it has nothing to do, or inventing a replacement that does not correspond to any real device. Three sanity guards catch these before anything touches the corrections file:

  1. The misheard words must appear verbatim in the transcript. If the model says dying room -> dining room for the transcript "I have the permissions", the word "dying" is not in that transcript — rejected.
  2. The replacement must match a known entity name or a recognised action word. "Good night -> Turn off the lights" fails this: "Turn off the lights" is not an entity name and is not in the action word list — rejected.
  3. The misheard phrase must be at least two words and not the entire transcript. Phonetic misrecognitions affect part of a sentence — a room name, a verb. If the "misheard" phrase is the whole input, it is either correctly transcribed or too short to correct safely — rejected.

These three guards have caught every hallucination we have seen in practice. The correction only lands if the model identifies something that is actually in the transcript, actually maps to a real device, and is a partial phrase rather than the whole utterance.

The full script is at whisper-corrections-cron.py in the repo. It is deployed at /home/ctf/homeassistant/whisper_corrections_cron.py and runs under cron:

0 2 * * * /usr/bin/python3 /home/ctf/homeassistant/whisper_corrections_cron.py \
  >> /home/ctf/homeassistant/whisper-logs/corrections_cron.log 2>&1

Each run also appends to corrections_review.log — a permanent audit trail of what was proposed, what was added, and what was rejected. If a correction starts causing problems, the log shows exactly when it was added and what the model said at the time.

Is this the right long-term solution? Probably not. A larger model with a longer context window would classify more accurately and not need sanity guards to catch its own hallucinations. But the right question for this hardware is not "what would work best?" — it is "what works reliably within these constraints?" The command set is small, the error rate is low, and the nightly window is the only time when doing a 2-second LLM call per error row is completely free. For a household running a limited number of daily voice commands, this is the right tradeoff.

What's Next: Public LLM as a Fallback

The local intent recognition handles the commands I know about. There will always be a long tail — a phrasing I did not anticipate, a new device someone asks about, a question rather than a command.

For the long tail, I plan to use a public LLM as a fallback rather than the local hailo-ollama. The key insight from this whole journey: what I needed was determinism, not intelligence. A small local model with enough context to be accurate introduces latency that defeats the purpose. A public cloud LLM for the rare cases that slip through local matching is a much better tradeoff — the latency is acceptable precisely because it is the exception, not the rule.

The important thing is that the main components stay local: STT on the Hailo NPU, custom sentence matching in HA, corrections in the proxy. None of these depend on a network connection. Busy servers and congested networks do not affect whether your lights turn on. The cloud LLM fallback only touches the long tail — novel requests rather than urgent device control — so the occasional latency there is acceptable.

That said, if I had a beefier local machine, I would still want a local LLM. A larger model with more memory headroom would not hit the same accuracy/latency tradeoff that killed it on the Pi. The Raspberry Pi is the wrong host for LLM inference; that does not mean local LLMs are the wrong answer in general.

Using the Published Images

After getting everything working, we packaged both components as public Docker images so that other people do not have to repeat the same painful discovery process.

The images are on Docker Hub:

  • canthefason/hailo-ollama — the proxy + runtime, ready to run in front of hailo-ollama
  • canthefason/wyoming-hailo-whisper — Wyoming protocol server wrapping the Hailo NPU Whisper encoder

There is an important design decision worth explaining here. The hailo-ollama binary itself and the HailoRT native library (libhailort.so) are Hailo's proprietary software. We cannot redistribute them inside a public Docker image. So the images contain only our code — the proxy, the Wyoming server, and their runtime dependencies. The Hailo binaries are bind-mounted from the Pi host at runtime.

This means the images are freely publishable, and you can upgrade HailoRT on the Pi without rebuilding the Docker image — just update the version suffix in the bind-mount path. The tradeoff is that you need HailoRT installed on the Pi host first, which is handled by 01-install-hailo-voice-assistant.sh.

The repo includes a fully annotated minimal compose file that covers only the voice assistant services:

docker-compose.voice-assistant.yml

To get started, there is one manual step before you run anything: the HailoRT packages are not publicly downloadable — Hailo requires a free developer account. Register at hailo.ai/developer-zone, download the two .deb files for HailoRT 5.3.0 (the runtime package and the PCIe driver), and copy them to your Pi's home directory:

scp hailort_5.3.0_arm64.deb hailort-pcie-driver_5.3.0_arm64.deb pi@rpi.local:~/

Once those are on the Pi, clone the repo and run the install scripts:

git clone https://github.com/canthefason/hailo-voice-assistant
bash 01-install-hailo-voice-assistant.sh   # sets up Pi, HailoRT, base stack
bash 02-install-hailo-ollama-npu.sh        # installs hailo-ollama with NPU + proxy

The scripts handle everything else from there — DKMS kernel module, Python bindings, Docker images, model download, and compose wiring.

Key Takeaways

The Hailo AI HAT+ delivers on its performance promise. The NPU is real and the latency improvement over CPU inference is substantial. But the software ecosystem is still young, and the path from "hardware that works" to "voice assistant that replaces Alexa" involved several pivots that were not obvious at the start.

The most important lesson: for a voice assistant handling a known set of home automation commands, what you need is determinism, not intelligence. A local LLM small enough to run on constrained hardware is non-deterministic by nature — you can improve accuracy by giving it more context, but more context means higher latency, and at some point you have a 30-second voice assistant that is worse than what you started with.

The right architecture turns out to be: fast local STT, deterministic local intent matching for known commands, corrections for the imperfect transcription cases, and a cloud fallback for the long tail. The local components handle the latency-sensitive part reliably. The cloud component only handles the edge cases where a bit of latency is acceptable.

The things that will trip you up most if you try to replicate this:

  1. The JSON sanitisation bug in hailo-ollama — both layers of it. Test with real HA system prompts, not synthetic ones.
  2. The blob storage path format — underscore not colon, singular blob not blobs.
  3. VDevice sharing — do not forget HAILO_VDEVICE_GROUP_ID in both containers.
  4. Model installation order — always stop hailo-whisper before installing a model.
  5. Use prefer_local_intents: true and write custom hassil sentences for anything you say regularly. Use a public LLM (or Claude) to generate the full set of surface forms — it is much faster than writing them by hand.
  6. Enable STT logging early. The log tells you what your household is actually saying and where transcription is failing. That data is how you improve the system over time.

The code and install scripts are on GitHub: github.com/canthefason/hailo-voice-assistant

One last honest note: Alexa is still plugged in, but barely. The daily commands — lights, shades, heat pump — are handled locally now, and reliably. The gap between this and a full Alexa replacement is small. More importantly, when something does not work, I know exactly where to look: the STT log shows what was heard, the HA trace shows whether a sentence matched, the proxy log shows what went to the LLM. With Alexa, that kind of visibility did not exist. Debugging was guesswork. That alone makes this worth it.
